Platform Vision — v0.2
Civic Intelligence Platform — all of society, all of India
This is no longer just a water body encroachment tracker. It is a civic intelligence system that observes, classifies, tracks, and narrates every meaningful change across all dimensions of public life — starting with Hyderabad, scaling to India.
🌏 Geographic ambition

Phase 1: Hyderabad — build the model, validate the approach, accumulate ground truth data.

Phase 2: Telangana state — replicate with state-level administrative data.

Phase 3: All of India — parameterize the system by state, district, and administrative unit.

The architecture must be designed for Phase 3 from Day 1. Hyderabad is the pilot, not the ceiling.

🎯 What "intelligence" means here

Not just data collection. The system must:

• Detect events autonomously (satellite, sensors)

• Classify and route without human intervention

• Build and maintain the story of every area over time

• Improve its own accuracy as evidence accumulates

• Know what it doesn't know and flag it

CTO note — scope discipline: This is now a large system. The instinct to track everything simultaneously will kill it. Research all domains first (see Domain Coverage), then ruthlessly prioritize which 3 domains to instrument in v1. Infrastructure + Land + Environment. Everything else is v2. Breadth kills momentum.

System Architecture v2
Google-scale patterns — how they do it
Google Earth Engine, BigQuery, and Bigtable solve the exact data bottleneck this system will face. We replicate those patterns using open-source and managed equivalents. The key insight: separate your write path from your read path.
📐 How Google handles this at scale (and how we replicate it)

Google's civic and satellite intelligence systems use three key patterns: (1) tiered storage by access frequency, (2) CQRS — separate write and read paths, and (3) pre-computed materialized views for expensive queries. We replicate all three.

📊 Storage tiers explained

Hot (Redis + CDN): Cached ward summaries, recent reports, active task board. Sub-100ms reads. Invalidated on write.

Warm (Postgres OLTP): All operational data — reports, tasks, evidence chains, contributors. Indexed for transactional queries.

Cold (ClickHouse OLAP): Historical time-series, analytics, monitoring metrics, ML training datasets. Columnar, fast aggregations.

Archive (Object Storage): Raw satellite images, photos, historical GeoJSON. Never queried directly — read by pipelines.

🔄 CQRS pattern

Command side (writes): All writes go through an event stream. Supabase Realtime handles low-volume events. As volume grows, Kafka replaces it. Events are immutable — never update an event, only append.

Query side (reads): Pre-computed views for every dashboard widget. The map never runs a live aggregate query — it reads a cached GeoJSON layer updated every 15 minutes.

Why this matters: Every bottleneck you mentioned comes from mixing OLTP and OLAP on the same Postgres instance. CQRS prevents this from day one.

Migration path: Start with Supabase Postgres for everything (acceptable for Hyderabad scale). When ward report queries start taking >2s, add ClickHouse as the OLAP layer. When event volume exceeds ~1k/day, add Kafka. Don't add these prematurely — the CQRS pattern means you can swap components without breaking the rest.

Data at Scale
Iteratively expanding schema — designed for growth
The schema must support adding new domains without breaking existing ones, and every piece of evidence must link back to the original issue it concerns. Think of each civic area as an accumulating story, not a static record.
🗂️ Area Story model — the central abstraction

Every geographic area in Hyderabad gets an area_story — an append-only record of everything that happened there. A story is the unit of civic intelligence. Reports, satellite observations, official documents, resolutions — all hang off the story.

-- Every area in Hyderabad (ward section / block)
area_stories(
  id, geo_id, ward_id, district,
  domain: 'infrastructure'|'land'|'environment'|'education'|...,
  status: 'active'|'resolved'|'monitoring'|'disputed',
  narrative_summary,    -- AI-generated, updated as evidence accumulates
  confidence_score,     -- composite trust score across all evidence
  opened_at, updated_at, resolved_at
)

-- All evidence links to a story
evidence_items(
  id, story_id,
  type: 'citizen_report'|'satellite_obs'|'govt_document'|'ai_inference'|'official_response',
  source_id,            -- FK to reports / satellite_alerts / documents
  contributor_id,
  trust_weight,         -- based on contributor tier + source type
  captured_at,
  payload_url           -- photo, GeoJSON diff, PDF, structured JSON
)

-- Domain-specific metadata (extensible via JSONB)
domain_metadata(story_id, domain, metadata JSONB)
-- e.g. road: { "damage_pct": 65, "road_class": "arterial", "last_maintained": "2021" }
-- e.g. lake: { "area_1990_ha": 12.4, "area_current_ha": 3.1, "encroached_pct": 75 }
Key design: domain_metadata uses JSONB so adding a new domain (agriculture, health) never requires a schema migration. The core area_stories and evidence_items tables are stable forever. New domains only add new rows with new JSONB shapes.
Road repair scenario (from your feedback): A story opens for the damaged road section. Evidence accumulates: citizen photos (week 1), satellite diff (week 2), traffic complaints (ongoing), official repair schedule (official response). A 6-month SLA timer starts. Each new citizen photo auto-links to the same story as a new evidence item. AI updates the narrative summary weekly: "Road damage confirmed by 14 citizen reports and 2 satellite observations. Official repair scheduled for Q3. No visible progress as of [date]."

Evidence System
Every claim needs a chain. Every chain tells a story.
The Evidence Manager is the core of the system's credibility. It organizes all evidence for an area story, weights it by source trust, and generates a running narrative that gets updated as new evidence arrives.
⚖️ Evidence weighting

Not all evidence is equal. Each item gets a trust_weight based on source type and contributor tier.

Satellite observation: 0.9 (highest — objective)

Official govt document: 0.85 (high — authoritative)

Verified contributor photo: 0.75

NGO report: 0.70

Anonymous citizen: 0.35

Story confidence = weighted average across all evidence items.

📖 Story states

Active: Issue confirmed, unresolved, accumulating evidence.

Monitoring: Claimed resolved, but system keeps checking satellite for 90 days.

Disputed: Conflicting evidence, needs human review.

Resolved: Satellite + citizen confirmation of resolution. Story archived but never deleted.

False Positive: Flagged as inaccurate after review. Logged for anti-gaming training.

AI Pipeline
Continuously trained, multi-modal, data flywheel
The system improves as it accumulates data. Satellite observations + citizen ground truth + official records combine into a training corpus that no single source has alone. This is the moat.
🌐 Multi-modal data stitching — telling the story of every area

Each area story in Hyderabad accumulates evidence across modalities. The AI layer must combine these into a coherent narrative:

Temporal: Landsat archive gives you 1972–present. For every area, you can compute what it was, what changed, and when. This is the unique capability no other system has — a 50-year timeline for every block in Hyderabad.

Spatial: Satellite observations are rasters (pixels). Citizen reports are points. Government boundaries are polygons. The system must fuse all three into a coherent representation per area.

Cross-domain: A lake encroachment is also a land record violation, also an environment story, also potentially a health story (reduced drainage). The AI must recognize domain overlap and link stories.

Learning angle — Geospatial ML: Stitching multi-modal geospatial data is the cutting edge of earth observation AI. The companies doing this (Descartes Labs, Orbital Insight, Planet) are hiring engineers who understand both the remote sensing domain and the ML stack. This project puts you on that path.

Classification Engine
Smart retries, threshold-based routing, human-in-the-loop
No single classifier is always right. The engine tries multiple approaches in sequence, switches when confidence is too low, and sends to human review only when it must — without flooding that queue.
🧠 Smart human review queue

The human review queue must never become a flood. Rules:

Smart batching: Group similar low-confidence reports together. A reviewer sees 10 pothole photos at once, not one at a time — batch decisions are faster and more consistent.

Domain routing: Infrastructure reports → infrastructure reviewer. Environment → environment expert. Don't mix.

Queue cap: Max 50 items in review at any time. If queue is full, new low-confidence items wait rather than overflow. This forces the system to improve Stage 1 + 2 accuracy.

📏 Per-domain thresholds

Different domains have different acceptable confidence thresholds. Higher impact → stricter threshold:

Land encroachment: ≥ 0.85 (high legal stakes)

Infrastructure damage: ≥ 0.75 (safety relevance)

Environment change: ≥ 0.70 (slower moving)

Construction progress: ≥ 0.60 (informational)

Thresholds are configurable and should be tuned monthly based on false positive rates per domain.

Contributor System
Tiered trust, incentives, and the monthly questionnaire
Contributors are not equal. The system must separate knowledgeable, reliable reporters from casual or bad-faith actors — and do so continuously, not just at registration.
📝 Monthly questionnaire — maintaining Verified status

Verified status is not permanent. It is maintained monthly via a personalized domain-specific quiz. The quiz tests:

• Domain knowledge (identify problem types from photos — same task Claude Vision does)

• Local knowledge (which department handles what in Hyderabad)

• Recent activity review (assess their own past 30 days of submissions)

Minimum passing score: 75%. Below threshold → downgraded to Registered. Status shown publicly on profile. This creates a natural quality filter without banning anyone.

🏆 Incentive system

Public leaderboard per ward — top contributors by verified report count, accuracy score, and domain coverage. Ward-level recognition.

Ward health badge — contributors in well-covered wards get "Ward Guardian" badge. Visible on profile.

Report outcome notifications — "Your report from March is now Escalated T2, 340 people have seen it."

Public forums — ward-specific discussion threads anchored to area stories. Contributors debate, add context, flag disputes.

🏛️ NGO integration

NGOs are treated as organizations, not individuals. The NGO portal allows:

• Bulk evidence submission with organization signature

• Access to aggregate ward data for their coverage areas

• Dashboard showing their submitted evidence's impact (stories influenced, escalations triggered)

• Priority in human review queue (their disputes are reviewed first)

NGO verification: basic org registration check + 30-day probationary period.

Security & Anti-Gaming
Defense in depth — adaptive, not static
Bad actors will find new ways to game the system over time. The defense must be layered so no single bypass defeats it, and must learn from each new attack vector. Every false positive flagged becomes a training signal.
🔁 Adaptive defense model

Security rules are not static config. They are a living ruleset maintained in the database and updated without code deploys.

When a new gaming pattern is discovered, it gets logged as an attack_pattern record with: pattern description, detection logic, affected layer, and date discovered. Monthly security review produces a "new attack vectors" report alongside the product report.

The AI filter (Layer 3) is retrained quarterly on accumulated false positive + confirmed fake report data.

⚖️ Appeal system

False positives happen. Every rejected report gets:

• A reason code (not just "rejected")

• An appeal link (valid for 7 days)

• A reviewer assignment if appealed

Successful appeals: report goes through human review queue, contributor reputation restored. High appeal success rate signals a bad rule, not bad contributors. Flag it for review.

CTO note: Don't over-engineer security at v1. Start with Layers 1–3 only. You won't have coordinated attacks until you have enough users to attract them. Add layers as you grow and as new attack vectors appear. The architecture supports adding layers — the key is having the logging in place from day one so you can see what's happening.

Monitoring System
Technical · Product · Impact — monthly + on-demand
The system monitors itself. Monthly reports are auto-generated across three perspectives. Any report can also be triggered on demand. Anomalies trigger alerts, not just scheduled reports.
⚙️ Technical report

Pipeline health: Satellite pull success rate, GEE export failures, AI classification throughput.

AI accuracy: Confidence distribution, false positive rate per domain, human review queue depth and age.

Infrastructure: DB query latency P95, error rates, storage growth rate, API call costs (Claude API spend).

Security: New attack patterns detected, rejection rate by layer, appeal success rate.

📦 Product report

Contributors: Active per tier, new verified this month, quiz pass/fail rates, NGO submission volume.

Coverage: Wards with ≥1 report, wards with zero coverage (highlight gaps).

Stories: Opened, closed, escalated, average age, resolution rate.

Domains: Report distribution by domain — where is the system being used vs. ignored?

📈 Impact report

Detection: Area stories opened by source (satellite-only, citizen-only, combined).

Resolution: Issues confirmed resolved, average days to resolution by domain.

Escalation: Complaints drafted, T2/T3 escalations triggered.

Data quality: Evidence items per story, confidence score distribution, false positive trend.

🔔 Anomaly detection — what triggers an immediate alert

Monthly reports are informational. These trigger immediate alerts (email + dashboard banner):

• AI classification confidence drops below 0.60 average for any domain (model degradation)

• Human review queue exceeds 50 items (pipeline bottleneck)

• Satellite pull fails for 7+ consecutive days (GEE issue or quota hit)

• Report submission rate spikes 10x in any ward within 2 hours (coordinated attack candidate)

• API cost exceeds monthly budget threshold (Claude Vision spend alert)

On-demand reports: Any authorized user can trigger a report via the monitoring dashboard for any date range, geographic scope (ward / district / city), and report type. Reports are pre-computed from ClickHouse materialized views — generation should be <5 seconds for any scope. No real-time DB aggregation on the read path.

Gaps & Open Questions
What we still don't have answers for
Updated for v0.2 scope. Some gaps are research gaps (we need to find the answer). Some are design gaps (we've chosen not to decide yet). Some are data gaps (the answer depends on what we find in the real world).
G1 — Domain scoring (research gap): We listed 8 domains. We haven't scored them yet by data availability, classification difficulty, civic impact, and legal risk. This scorecard must be done before v1 scoping. Without it, we'll build for the wrong 3 domains first.
G2 — Resolution detection (design gap): We still have no automated way to confirm an issue is resolved. Satellite diffs work for large-scale changes. For road potholes, drainage, or small construction — we don't have an answer. Options: second citizen report, periodic satellite diff, scrape government portals. None are clean. This is the biggest unsolved loop in the system.
G3 — Historical boundary data (data gap): HMDA lake and land boundaries from the 1990s may not be digitally available. Quality unknown until we actually try to obtain them. If the data isn't there, the historical encroachment story doesn't work. Research task: attempt to obtain HMDA historical GIS data before committing to the encroachment detection feature.
G4 — ClickHouse vs. BigQuery (design gap): We've recommended ClickHouse (self-hosted) for OLAP. BigQuery (managed, pay-per-query) is easier to start with. The choice depends on expected query volume and budget. At Hyderabad scale, BigQuery sandbox (free) is probably right. At India scale, ClickHouse becomes cheaper. Defer this decision to when you actually need the OLAP layer.
G5 — India expansion data model (design gap): Hyderabad's administrative structure (GHMC, HMDA, Mandal, Ward, Block) is Telangana-specific. Every Indian state has different administrative hierarchies. The schema must abstract these into a generic hierarchy (L1–L5) rather than hardcoding Hyderabad terminology. Design this from day one or the migration to another state will be a full rewrite.
G6 — Expert reviewer recruitment (people gap): The classification engine relies on expert reviewers to close the human review queue and provide training labels. We have no plan for recruiting domain experts (urban planners, environmental scientists, infrastructure engineers) who would volunteer time. Without them, the human review queue backs up and the model doesn't improve. This is a people problem, not a technical one.

Milestones — research first, then build
The first milestone is now research, not code. This is the right call given the expanded scope — building before researching produces the wrong system.
Milestone 0 — 2–3 weeks (NOW)
Domain research + scorecard
Score all 8 domains by data availability, classification difficulty, civic impact, legal risk. Attempt to obtain HMDA historical GIS data. Choose the 3 domains for v1. This is a research task — produce a document, not code.
Milestone 1 — 2–3 weeks
Core schema + citizen report app
Supabase schema with area_stories + evidence_items abstraction. Generic admin hierarchy (L1–L5, not hardcoded to Hyderabad). Citizen report form. OpenFreeMap. Pins appear on map. No AI yet.
Milestone 2 — 2 weeks
Classification engine Stage 1 + monitoring skeleton
Claude Vision classifier with structured JSON output. Threshold-based routing. First version of monitoring dashboard (even if just a Supabase query). Domain-specific confidence thresholds configured.
Milestone 3 — 3–4 weeks
Satellite water/land layer + evidence chains
GEE Python pipeline for Sentinel-2. Evidence Manager linking citizen reports + satellite observations to the same area_story. AI narrative summary generation (weekly update). Evidence chain view in UI.
Milestone 4 — 3–4 weeks
Contributor tiers + security layers 1–3
Registration + verification flow. Monthly questionnaire system. Reputation scoring. Rate limiting, geo-fencing, AI duplicate detection. Contributor leaderboard per ward.
Milestone 5 — 3 weeks
Historical change detection + task board + alerts
Landsat archive composites at 1990/2000/2010/2024. Change polygons. Department task board. Escalation tiers with complaint draft generator. Monthly auto-report (technical + product + impact).
Milestone 6 — Research track
Historical water recovery model
Fine-tune SegFormer on HMDA lake boundaries + historical Landsat. This is the research paper. Add ClickHouse as OLAP layer when query latency starts to hurt.
Milestone 7 — Scale
Second domain + India parameterization
Add Agriculture domain (NDVI farmland health). Validate generic admin hierarchy works for another Telangana district. Prove the system is portable before claiming India scope.
Most important constraint: Don't start Milestone 1 before Milestone 0 is done. The schema in M1 depends on domain choices made in M0. Building the wrong schema is more expensive to fix than the 2–3 weeks M0 costs.
